# 합성 Traceability + 공정 원인 데모 설계 (ontorag / ontorag-flow)

> 목적: ontorag(추론) + ontorag-flow(행동)의 부품 조합이 end-to-end로 맞물리는지를,
> 합성 제조 데이터로 검증한다. 무게중심은 **부품 추적(범위 좁히기) → 공정 인과(원인 규명)** 의 직렬 흐름.

---

## 0. 전제와 포지셔닝

- **합성 데이터는 차선이 아니라 적극적 선택.** 부품→로트→공급사→완제품의 관계형 추적 데이터는 기업 영업비밀이라 공개본이 거의 없음. 합성이면 인과·관계 구조를 직접 통제하고, ground truth를 알고 있어 검증이 자명함.
- **데모의 주장 범위를 정직하게 한정.** "실공장에서 검증됐다"가 아니라 **"실데이터를 이 스키마로 모델링하면 이렇게 추적·규명된다"는 방법론 증명**으로 포지셔닝.
- **원칙: 스키마(온톨로지)가 먼저, 데이터는 그 스키마를 따라 생성.** ontorag의 "온톨로지 = source of truth" 철학을 데이터 생성 단계부터 지킨다. 데이터부터 만들면 스키마가 데이터에 끌려다님.

## 1. 무게중심 / 시나리오 골격

한 시나리오 안에서 두 단계가 **직렬**로 흐른다 (두 기능이 경쟁하지 않고 보완).

1. **불량 완제품 집합** 발생
2. **(그래프 순회 / traceability)** 의심 로트·공정으로 범위 좁히기 — *전단계*
3. **(인과 추론 / L3)** 그 공정의 어느 조건이 원인인지 + 개입했다면 어땠을지 — *본론*

하이라이트: 불량 원인이 **부품 탓인지 공정 탓인지** 즉답이 안 되고, 인과 추론(do_query)으로 갈라야만 답이 나온다. 단순 상관(불량 제품에 공급사 B가 많더라)으로는 불가 → **L3의 존재 이유를 증명하는 시나리오**.

## 2. 온톨로지 스키마 (공정 포함)

### 엔티티 타입
- `Supplier`
- `Lot`
- `Component`
- `ProcessStep` — 공정 단계 정의 (가공 / 조립 / 검사직전처리)
- `ProcessRun` — 특정 부품이 특정 공정을 통과한 인스턴스 (공정 조건값을 보유)
- `Product`
- `QCResult` — Pass / Fail

### 관계
```
Supplier --supplies--> Lot
Lot --hasComponent--> Component
Component --processedBy--> ProcessRun
ProcessRun --atStep--> ProcessStep
ProcessRun --produces--> Product
Product --hasQC--> QCResult
```

### 핵심 설계 결정: 공정 조건은 **이산 상태**로 표현
- pgmpy 기반 L2/L3는 이산 변수를 다룸. 연속 센서값(온도/압력)을 그대로 넣지 말 것.
- `ProcessRun`의 조건을 구간화된 상태로: 예) `temperature = Normal | High`, `pressure = Normal | Low`.
- **합성이라 처음부터 이산으로 설계 가능** → discretize 전처리 불필요. (실데이터였다면 구간화 단계가 별도로 필요했을 지점.)

### 제약 (L1 논리 레이어가 작동할 근거)
- 모든 `Product`는 최소 1개 `Lot`을 (경로상) 포함한다.
- `Lot`은 정확히 1개 `Supplier`에서 온다.
- 모든 `ProcessRun`은 정확히 1개 `ProcessStep`에 속한다.
- (추가 후보) 모든 `Product`는 정확히 1개 `QCResult`를 가진다.

## 3. 인과 모델 (= 데이터 생성 규칙 = 추론 모델)

같은 인과 모델로 **데이터를 생성하고 동시에 추론**한다 → ground truth가 자명.

### 두 개의 인과 경로 + 상호작용
- **부품 경로:** `Supplier품질 → Lot품질 → Component품질 → Product불량`
- **공정 경로:** `공정조건(예: 온도 High) → Product불량`
- **상호작용:** 정상 부품이라도 공정3 온도가 High면 불량 / 나쁜 부품은 정상 공정에서도 일부 불량.

이 구조가 그대로 **베이지안 네트워크**가 된다. 각 화살표에 조건부 확률(CPT) 부여 → L2(확률)·L3(반사실)가 답할 대상.

### 동어반복 함정 회피 (중요)
단순히 "공급사 B = 불량"으로 심으면, 추적 성공이 시스템의 똑똑함이 아니라 데이터가 뻔한 것. **노이즈를 의도적으로** 넣어 "자명하지 않은데 맞히는" 그림을 만든다:
- 공급사 B의 결함률은 베이스라인보다 높지만 100%는 아님 (로트별 편차).
- 특정 로트(예: #47)는 추가로 오염.
- 조립 공정에서도 낮은 확률로 우연 불량 혼입.
- Product 불량 = 위 원인들의 **조합**으로 결정.

## 4. 복잡도 통제 — 처음엔 의도적으로 작게

전체 구조는 위와 같이 설계하되, **첫 구현은 최소 인과 모델**로 시작.

- 공정 단계: **3개** (예: 가공 → 조립 → 검사직전처리), 각 단계 이산 조건 1개씩.
- 공급사: **3~5개**.
- 베이지안 네트워크 노드: **6~8개 이내**로 유지 → CPT를 손으로 채울 수 있고 추론 결과도 직관적으로 검증 가능.
- 노드가 많아지면 CPT 폭발 + 데모가 블랙박스처럼 보임. **작게 시작 → 인과가 깨끗이 도는 것 확인 → 키우기.**

**확정값(추천):**
- 공정 3단계, 각 단계 이산 조건 1개.
- 결함 원인 2종: 공급사발 부품 결함 1 + 공정조건 결함 1, + 둘의 상호작용 1.
- 데이터 규모(초기): 공급사 5, 로트 50, 완제품 300.

## 5. 작업 단계 (순서 고정)

### 1단계 — 온톨로지 스키마 설계 (가장 먼저)
- 엔티티/관계/제약을 OWL(Turtle)로 정의. 위 §2가 초안.
- 제약이 L1(논리)의 작동 근거.

### 2단계 — 인과/결함 모델 설계
- §3의 베이지안 네트워크 구조(노드·엣지·CPT) 확정. 데이터 생성 규칙이자 추론 모델.

### 3단계 — 데이터 생성기 작성
- 2단계 인과 모델을 코드로 구현 → 인스턴스 생성.
- 출력: **RDF/Turtle** (ontorag가 바로 적재). 동시에 **정답표**(어느 로트/공정이 진짜 원인인지) 별도 저장 → 검증용.

### 4단계 — ontorag 단독 검증 (flow 없이 먼저)
- Fuseki에 적재, MCP 툴로 추적 확인.
  - "불량 완제품들의 공통 로트는?" (다중 홉 순회)
  - "공급사 B를 정상으로 개입했다면 불량률은?" (`do_query`)
- 정답표와 대조. 막히면 flow 문제가 아니라 데이터/스키마 문제로 범위가 좁혀짐.

### 5단계 — ontorag-flow로 결정 흐름 얹기 (조합 검증 본론)
- 시나리오: 불량 급증 케이스 발생 → 엔진이 의심 로트 추천 → 운영자 승인 → write-back으로 "로트 #47 격리" 기록 → provenance에 결정 경로 기록.
- `process YAML`로 흐름 정의 (`tutorials/first-case`, `authoring-process` 가이드 활용).
- `counterfactual replay`("그 로트를 안 격리했다면")까지 돌면 조합이 완전히 닫힘.

## 6. 부품 ↔ 데모 매핑 (조합 완전성 체크)

| ontorag / ontorag-flow 부품 | 이 데모에서 쓰이는 지점 |
| --- | --- |
| L0 저장소 (Fuseki/Neo4j/FalkorDB) | 합성 Turtle 적재 |
| L1 논리 (제약, subClassOf, transitive) | traceability 경로 보장, "공통 로트" 순회 |
| L2 확률 (pgmpy posterior) | "이 조건 조합에서 불량 확률" |
| L3 반사실 (do_query, counterfactual) | "공급사/공정에 개입했다면" 원인 분리 — **본론** |
| 결정 엔진 (6종) | 의심 로트/공정 추천 |
| Case + 상태머신 + saga | 불량 케이스 lifecycle, 격리 조치 |
| Provenance (PROV-O) | 결정 경로 기록, forensic replay |
| write-back (AssertTriple/RetractTriple) | "로트 격리" 사실 갱신 → 폐루프 |

## 7. 리스크 / 주의

- **합성 함정:** 결함 규칙이 단순하면 동어반복. §3 노이즈 필수.
- **CPT 폭발:** 노드 수 통제(§4). 손으로 채울 수 있는 규모 유지.
- **연속값:** 공정 조건은 처음부터 이산 상태로(§2). discretize 회피.
- **도메인 전이:** 이 인과 모델링 사고는 회사 공급망 충격 시뮬레이션(Leontief IO)과 구조가 닮음 → 사고 패턴 전이. 단 IP 분리 원칙 유지(공개 데이터·개인 repo).
